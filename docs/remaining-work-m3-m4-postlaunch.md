### Remaining code gaps that block M4 acceptance criteria

These are acceptance-criteria items from Section 2 of the spec that are **not yet implemented**:

#### E. Abuse / Reliability — partially incomplete

| Item | Status | Notes |
|---|---|---|
| IP/ASN velocity checks | ❌ Not implemented | Spec requires this alongside the signed device identifier. Mitigation: Vercel's built-in rate limiting can be used at the edge via middleware, or Cloudflare Turnstile for challenge escalation. |
| Challenge escalation (Turnstile or equivalent) | ❌ Not implemented | Listed as a required abuse control. Optional Turnstile integration on the guest chat path would satisfy this. |
| Signed device identifier | ❌ Not implemented | Currently device identity is the `ata_anon_id` cookie (random UUID). A "signed" identifier would require a server-issued token. Minimum viable: promote the anon cookie to a server-set `HttpOnly` cookie generated by a Next.js API route. |
| Fallback UX for provider outage | ⚠ Partial | Error messages are surfaced in chat bubbles but there's no dedicated fallback UI or retry logic. |

#### D. Security — one gap

| Item | Status | Notes |
|---|---|---|
| BYOK persistent storage "encrypted at rest client-side" | ❌ Not implemented | Spec says persistent mode uses encryption at rest. Current implementation stores the raw key string in `localStorage`. Minimum viable: `SubtleCrypto.encrypt()` with a key derived from the Clerk userId or a user-supplied passphrase. |

#### A. UX — design parity

| Item | Status | Notes |
|---|---|---|
| Design asset parity check | ❌ Not done | Spec requires parity verification against all 5 draw.io assets before launch: `chat-interface-mockup.drawio`, `philosopher-browser.drawio`, `history-page.drawio`, `bookmarks-page.drawio`, `settings-page.drawio`. Use `drawio-mcp-server` to diff. |
| Guest streaming responses | ⚠ Non-streaming | Guest mode uses the Convex action path (no streaming). Spec says "streaming response works on mobile and desktop". Guest users see a thinking animation but no live tokens. True guest streaming requires a Next.js proxy route using `GROQ_API_KEY`. |

## Post-Launch Backlog (V1.1 → V2)

Ordered by spec priority. None of these are M3/M4 blockers.

1. **Guest streaming** — route guest inference through `/api/chat/stream` using the server-side `GROQ_API_KEY`, giving guest users live token delivery. (Deferred from M1 because it required exposing the Groq key in the streaming path.)

2. **IP/ASN velocity checks + Turnstile** — add Vercel edge middleware or Cloudflare for IP-level rate limiting and bot challenge escalation.

3. **BYOK persistent encryption** — implement `SubtleCrypto`-based encryption for persistent key storage mode, satisfying the spec's "encrypted at rest" requirement.

4. **Signed device identifier** — server-issued `HttpOnly` cookie replacing the client-generated anon UUID.

5. **Provider abstraction expansion** — additional BYOK providers, model picker per-key, custom endpoint per-key stored in `KeyMeta`.

6. **Better eval coverage** — gold-set retrieval relevance checks, per-philosopher eval baselines, regression tracking.

7. **Indian schools integration** — new traditions with licensing-first workflow (V2 scope per spec).

8. **Stretch features** — debates, voice, daily digest, community sharing (V2+).
